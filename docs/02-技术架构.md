## 一、技术栈（已拍板）

| 层 | 选型 |
|---|---|
| 小程序端 | 微信原生（vtble 插件/转发/手机号组件强绑定）|

## 二、AI 餐食解析管线（项目心脏）

```
拍照 → 小程序直传 OSS(私有桶) → 后端取签名URL → 调 Chester 提供的国内视觉API
     → 结构化 JSON{食物,份量,kcal} → 合规过滤层 → 返回用户确认/修正 → 入库汇总
```

### AI 输出契约（强制 JSON）

```json
{
  "items": [
    {"name": "番茄炒蛋", "portion": "约1盘(300g)", "kcal_est": 180, "confidence": "high"}
  ],
  "remark": ""
}
```

### 合规过滤层（出口必经）

输出**只许食物事实**（名称/份量/能量）；prompt 明令禁止 + 代码层再过滤：医疗建议、减肥指导、"超标/不健康/少吃"评价、禁用词表词汇。能量一律标"估算值"。

### 失败兜底

照片**先落库**再异步解析；解析失败 → `ai_status=failed` → 用户从常见食物能量库手选或手填 kcal（食物库首发 300-500 条）。

## 三、数据模型（11 张表）

| 表 | 关键字段 | 说明 |
|---|---|---|
| `users` | id, openid, phone, nickname, gender, height_cm, target_weight_kg, advisor_id, group_tag, status | advisor_id=服务老师标签（单层）；group_tag=后台自由分组 |
| `advisors` | id, name, phone, note, active | 服务老师字典，仅后台维护，**不是登录账号** |
| `admin_accounts` | id, username, password_hash, role(admin/viewer) | 仅两档 |
| `consent_records` | id, user_id, type(privacy/health_data), version, granted_at, revoked_at | 同意留痕，支持撤回 |
| `weight_records` | id, user_id, weight_kg, measured_at, source(manual/photo/ble), is_morning, photo_key, supplier_record_id, raw_payload, **14 项体成分**（bmi/body_fat_pct/fat_kg/subcut_fat_pct/visceral_fat_level/water_pct/skeletal_muscle_kg/muscle_kg/bone_kg/bmr_kcal/protein_pct/body_age/health_score/fatty_liver_level）| 体成分均可空（蓝牙时由 raw_payload 归一化）；fatty_liver_level **存而默认不展示** |
| `meal_records` | id, user_id, meal_date, meal_type(breakfast/lunch/dinner/snack), photo_key, photo_kind(food/label), ai_status, ai_raw, note | 一次打卡一条 |
| `meal_items` | id, meal_record_id, food_name, portion_text, kcal, source(ai/user/label_ocr) | 当日汇总 = sum(kcal) |
| `daily_logs` | user_id+date 唯一, water_cups, water_cup_ml(默认250), bowel_count, bowel_status(smooth/normal/hard), sleep_hours, note | 选填快捷记录；可加可减 |
| `advisor_change_logs` | id, user_id, old_advisor_id, new_advisor_id, changed_by, changed_at | 归属调整留痕 |
| `data_shares` | id, owner_user_id, viewer_user_id, status, source(user_invite/company_assign), invite_code, created_at, revoked_at | (owner,viewer) 活跃唯一；移除即时生效 |
| `service_notes` | id, owner_user_id, author_user_id, ref_date, content, visibility(visible_to_owner 默认/team_only) | visible_to_owner=「老师的话」用户当日汇总展示 |
| `service_day_marks` | id, owner_user_id, viewer_user_id, date, mark(reviewed/flagged) | 服务者逐日归档/关注；(owner,viewer,date) 唯一 |
| `stage_records` | id, user_id, stage(baseline/periodic/final), taken_at, front_photo_key, side_photo_key, back_photo_key, tongue_photo_key, waist_cm, hip_cm, thigh_cm, whr(生成列), visible_to_advisors(默认true), note | 体型照三面+围度+舌诊；photo_key 全私有桶 |
| `health_reports` | id, user_id, type(biochem/physical/other), file_key, taken_at, note, visible_to_advisors | 生化/体检报告；仅存储展示，**系统不做任何医疗解读** |
| `notifications` | id, user_id, type, title, body, target(JSON), read_at, created_at | 通知类型 + 跳转 target |
| `onboarding_tasks` | user_id, task_key, done_at | S17 开启之礼 5 项进度 |
| `shipments` | id, user_id, recipient, phone, address, status, tracking_no, created_at | 礼品发货管理 |
| `media_download_logs` | id, operator_id, owner_user_id, scope(single/batch), file_keys/date_range, created_at | 下载审计 |
| `export_jobs` | id, operator_id, params, status, zip_key, expires_at | 异步导出任务 |

**每日汇总不建表**，用查询聚合：`总摄入 = Σ meal_items.kcal`；`晨重 = weight_records where is_morning`。

## 五、图片存储与下载架构

- **上传前小程序端压缩**：长边 ≤1600px、质量 80% → 单张 ~250KB；**剥离 EXIF**
- **缩略图不预存**：用 OSS 图片处理参数实时出图（列表 200px / 详情 800px / 原图按需）
- **生命周期自动化**（OSS 规则）：>90 天转低频 → >365 天转归档 → 留存期（24 个月）自动删除/匿名化
- **数据库只存 file_key**，图片永不进 DB
- **批量导出走异步**：服务端打 zip 放临时桶 → 24h 失效链接 → 写 `media_download_logs` 审计；仅 admin 角色

## 六、安全与密钥原则

- appsecret / DB 密码 / AI API key 只放服务器环境变量，**永不进 Git**
- 餐照/秤照/体型照/报告 全私有桶；后台看图也走短时效签名 URL
- 数据库每日自动备份；上线前做一次恢复演练
- 服务视图行级权限：`/service/*` 每个请求校验存在 active 共享，否则 403；服务者侧手机号脱敏、无导出接口
- 仓库/Issue/测试数据零真实隐私（假手机号段 138-0000-xxxx）
