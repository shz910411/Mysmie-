# CLAUDE.md · 项目宪法

> 任何 AI 窗口（PM / Dev / QA）启动时第一份必读文件。本文件压倒一切外部偏好。

## 一、项目本质

**迈思美轻体记录工具**：纯使用工具，无支付、无团队、无奖励。用户称重 + 拍照吃饭 + AI 解析能量 + 当日汇总 + 一键转发微信群。运营激励全在微信群，小程序只做"群里做不了的"。

## 二、Karpathy 4 守则（默认开启）

1. **先想后写**：假设要显式说出来；不确定就停下问；多解读时摆选项不默选。
2. **极简优先**：只写本任务的最小代码；不留扩展位；200 行能压 50 就重写。
3. **外科手术式改动**：每行 diff 都能追溯到当前 Issue；不顺手"优化"无关代码。
4. **目标驱动可验证**：开工前重读该 Issue 的"成功标准"，没写就退回 PM 窗补。

## 三、3 窗角色铁律

| 窗 | 干什么 | 禁区 |
|---|---|---|
| PM | 维护 docs/ + 排 Issue + 定成功标准 | 不写代码、不验收 |
| Dev | 从 issues.md 取一条 → 写代码 → commit | 不改需求、不自验收 |
| QA | 按成功标准真编译+真数据验收 | 不修代码、不放水 |

**跨窗通信只走 docs/ + git commit + issues.md，禁止口头**。Dev 想到新需求 → 回 PM 窗登记；QA 发现问题 → 回 Dev 窗派 Issue。

## 四、一条反传销铁律（永不破）

本小程序**永不**出现：奖励/积分/返利、团队层级展示、独立经销商后台、按"带来/服务多少人"的任何排名或展示；数据共享关系**永远扁平**（不构成上下级、不可层级穿透）；advisor 标签仅作公司内部服务分组，**任何数据不回流九洲福作计酬/晋级输入**。未来加激励功能前，必须先过合规评审。

## 五、合规口径（用户可见层硬约束）

- **轻体口径**：用户可见层（页面/按钮/推送/分享卡/图片内文字）一律"轻体"，**零出现**"减脂/减肥/瘦/燃脂/塑形"及一切极限词、医疗暗示词。
- **slogan**：核心 = 「吃饱也能瘦，吃对样子好，自律成良习」；**线上小程序可见层用「吃饱也能轻」**（一字之改）。
- **AI 输出**：✅ 允许搭配/份量/营养缺口反馈；🔴 禁医疗词、疾病关联、效果承诺、否定式评价、营养处方；所有 AI 文案走白名单模板，出口再过禁用词表；常驻"生活方式参考建议·非医疗/营养处方"声明。
- **能量**永远标"估算值"。
- **报告/舌诊照**：仅存储展示，**系统不做任何 AI 自动解读**。

## 六、关键开发约束

| 项 | 约束 |
|---|---|
| 真编译铁律 | wxml 必须微信开发者工具真编译过，不接受 node --check |
| 真实数据铁律 | 除蓝牙称重外，所有功能必须真实数据落库，禁纯 mock |
| AI 视觉 | 直接用 Chester 提供的旧版国内 API，不再做选型 PoC |
| 蓝牙称重 | 本地开发期可模拟器代替；上线走 vtble 插件，**手动/拍秤永远 P0 不可被卡死** |
| 隐私同意 | S2 两段式同意，健康数据**独立勾选+非默认+不捆绑** |
| 密钥 | appsecret/DB密码/AI API key 永不进 Git，只在服务器环境变量 |
| 图片 | 餐照/秤照/体型照/报告 全私有桶 + 短时效签名 URL；上传前剥离 EXIF |
| 共享 | 邀请码 24h 单次有效；移除即时生效；服务者只读+备注 |
